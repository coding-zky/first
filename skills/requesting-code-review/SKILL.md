# 代码审查

## 概述

对照实现计划和规范审查代码变更。按严重程度分类问题：关键问题阻塞进度，警告需要关注，建议可以稍后处理。

**开始时宣布：** "我正在使用 requesting-code-review 技能审查当前批次的代码变更。"

## 审查流程

### 1. 获取变更

首先确定要审查的内容：
- 自上次审查以来的 git diff
- 当前批次中修改的文件
- 相关的测试结果

### 2. 对照计划检查

逐项检查实现是否与计划一致：
- 是否创建/修改了计划中指定的所有文件？
- 每个功能是否按照规范的预期行为实现？
- 是否有计划外添加的内容？

### 3. 代码质量检查

- 命名：变量、函数、类名称是否清晰且有意义？
- 结构：代码是否遵循单一职责原则？
- 重复：是否有应该消除的重复代码？
- 边界情况：是否处理了空值、错误状态和异常？

### 4. 测试质量检查

- 测试覆盖：所有新功能都有测试吗？
- 测试质量：测试验证行为而非实现吗？
- 边界情况：测试覆盖了失败路径和异常吗？

## 问题严重程度

### 关键（阻塞进度）
- 代码与计划/规范有显著偏差
- 缺少关键功能的测试
- 安全漏洞
- 可能导致数据丢失或损坏的 bug

### 警告（应在继续前修复）
- 缺少边界情况测试
- 明显的性能问题
- 不一致的错误处理
- 重复代码

### 建议（可稍后处理）
- 命名可以更清晰
- 可以提取辅助函数
- 注释可以更详细

## 审查报告格式

```markdown
## 代码审查报告 — 批次 N

### 概述
- 审查文件: N
- 关键问题: N
- 警告: N
- 建议: N

### 关键问题
1. [问题描述] — [文件:行号]

### 警告
1. [问题描述] — [文件:行号]

### 建议
1. [建议描述] — [文件:行号]

### 结论
- 如果有关键问题：必须在继续前解决
- 如果没有关键问题：可以继续下一批次
```
