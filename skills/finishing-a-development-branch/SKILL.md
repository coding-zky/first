# 完成开发分支

## 概述

当所有实现任务完成时，进行最终验证、清理工作并呈现合并选项。

**开始时宣布：** "我正在使用 finishing-a-development-branch 技能完成并收尾此开发分支。"

## 收尾流程

### 1. 最终验证

在声明完成之前，验证所有内容：

- [ ] 所有计划任务已完成（检查 tasks.md 中的复选框）
- [ ] 所有测试通过（运行完整测试套件）
- [ ] 没有遗留的调试代码、console.log 或临时文件
- [ ] 代码已按照计划进行审查
- [ ] OpenSpec 变更已同步到规范目录

### 2. 清理检查

检查代码库中常见的在开发过程中积累的问题：

- 注释掉的代码 — 删除它
- 调试语句（console.log、print、debugger）
- 临时文件或未使用的导入
- 未使用的变量和函数
- 硬编码的值应该提取为配置

### 3. Git 状态检查

```bash
# 检查工作区状态
git status

# 检查未跟踪的文件
# 如果有应该忽略的文件，更新 .gitignore

# 检查提交历史
git log --oneline -10
```

### 4. 变更总结

生成变更总结：

```markdown
## 开发分支总结

### 概述
- 分支: `<branch-name>`
- 基于: `<base-branch>`
- 提交数: N

### 变更内容
- 新增文件: N
- 修改文件: N
- 删除文件: N

### 功能变更
1. [功能描述] — `[commit-hash]`
2. [功能描述] — `[commit-hash]`

### 测试
- 测试文件: N
- 测试用例: N
- 全部通过: 是/否
```

### 5. 呈现合并选项

向用户呈现选项：

1. **合并到主分支** — 直接合并
2. **创建 PR/MR** — 创建拉取请求进行审查
3. **保留分支** — 暂不合并，保留以备后用
4. **放弃分支** — 丢弃所有变更

### 6. 归档 OpenSpec 变更

如果启用了 OpenSpec：
- 将变更文件夹移动到 `openspec/changes/archive/YYYY-MM-DD-<变更名>/`
- 将 delta 规范合并到 `openspec/specs/` 中的主规范
- 提交归档变更

## 完成确认

最终向用户确认：

"开发分支 `<branch-name>` 已完成。所有 N 个任务已实现，N 个测试全部通过。变更已总结并准备合并。你希望如何处理？"
