# 测试驱动开发 (TDD)

## 概述

强制执行严格的 TDD 红-绿-重构循环。在编写任何实现代码之前编写测试，验证测试失败，编写最小实现使其通过，然后重构。

**开始时宣布：** "我正在使用 test-driven-development 技能，遵循红-绿-重构循环。"

## 红-绿-重构循环

### 红色阶段：编写失败的测试

1. 编写一个最小、聚焦的测试用例
2. 测试必须验证一个特定的行为
3. 运行测试并确认它失败
4. 如果测试通过而没有实现，说明测试有问题 — 修正测试

### 绿色阶段：编写最小实现

1. 编写刚好足够让测试通过的代码
2. 不要添加测试未覆盖的功能（YAGNI）
3. 不要担心代码质量 — 先让它工作
4. 运行测试并确认通过

### 重构阶段：改进代码

1. 消除重复代码（DRY）
2. 改进命名以提高可读性
3. 提取辅助函数和方法
4. 运行测试确保所有测试仍然通过
5. 只有在所有测试通过后才提交

## 测试反模式（禁止）

- **禁止** 编写测试后立即编写完整实现 — 总是先看测试失败
- **禁止** 跳过红色阶段 — 这是验证测试有效性的关键步骤
- **禁止** 在重构阶段添加新功能 — 新功能需要新的测试
- **禁止** 在测试失败时提交
- **禁止** 编写不验证任何行为的测试（没有断言的测试）

## 提交规范

```bash
# 每次红-绿-重构循环后提交
git add <changed files>
git commit -m "test: add failing test for <behavior>"
# 然后实现后提交
git commit -m "feat: implement <behavior> to pass test"
# 重构后提交
git commit -m "refactor: improve <behavior> implementation"
```

## 测试设计原则

- 测试行为，不测试实现
- 每个测试只验证一件事
- 测试名称描述预期行为
- 使用 Given-When-Then 结构
- 保持测试独立 — 测试之间不共享状态

## 提交前检查清单

- [ ] 所有测试通过
- [ ] 没有跳过的测试或 TODO
- [ ] 代码是最小实现（YAGNI）
- [ ] 没有重复代码（DRY）
- [ ] 提交信息描述清晰
